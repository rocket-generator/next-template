# 国際化（i18n）ガイド

Next.js + next-intl による多言語対応の実装ガイド。

## 特徴

- 日本語・英語に対応
- Server Components / Client Components どちらでも利用可
- 階層的メッセージ構造
- TypeScript による型安全性
- タイムゾーン: `Asia/Tokyo`

## セットアップ

### Next.js 設定

```typescript
// next.config.ts
import createNextIntlPlugin from "next-intl/plugin";
const withNextIntl = createNextIntlPlugin();
export default withNextIntl({ /* ... */ });
```

### ルーティング設定（`src/i18n/routing.ts`）

```typescript
import { createNavigation } from "next-intl/navigation";
import { defineRouting } from "next-intl/routing";

export const routing = defineRouting({
  locales: ["ja", "en"],
  defaultLocale: "ja",
  pathnames: { "/": "/" },
});

export type Locale = (typeof routing.locales)[number];
export const { Link, getPathname, redirect, usePathname, useRouter } =
  createNavigation(routing);
```

### リクエスト設定（`src/i18n/request.ts`）

```typescript
import { getRequestConfig } from "next-intl/server";
import { routing } from "./routing";

export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;
  if (!locale || !routing.locales.includes(locale as Locale)) {
    locale = routing.defaultLocale;
  }
  return {
    locale,
    timeZone: "Asia/Tokyo",
    messages: (
      await (locale === "en"
        ? import("../../messages/en.json")
        : import(`../../messages/${locale}.json`))
    ).default,
  };
});
```

### ルートレイアウト / Providers

`src/app/layout.tsx` で `getLocale()` / `getMessages()` を取得し、`src/app/providers.tsx` の `NextIntlClientProvider` に渡す。

## メッセージファイル構造

`messages/ja.json`, `messages/en.json` を階層的に整理する。

```json
{
  "Auth": {
    "signin": "ログイン",
    "email": "メールアドレス"
  },
  "Menu": {
    "Admin": { "users": "ユーザー" }
  }
}
```

## 使用方法

### Server Component

```typescript
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("Menu.Admin");
  return <div>{t("users")}</div>;
}
```

### Client Component

```typescript
"use client";
import { useTranslations } from "next-intl";

export default function SignInForm() {
  const tAuth = useTranslations("Auth");
  return <button>{tAuth("signin")}</button>;
}
```

## 高度な機能

### 動的な翻訳

```json
{ "welcome": "こんにちは、{name}さん！" }
```

```typescript
t("welcome", { name: "山田太郎" });
```

### Rich Text

```json
{ "description": "詳細は<link>こちら</link>をご覧ください" }
```

```typescript
t.rich("description", {
  link: (chunks) => <Link href="/details">{chunks}</Link>,
});
```

### 日付・数値フォーマット

```typescript
import { useFormatter } from "next-intl";
const format = useFormatter();
format.dateTime(now, "short");
format.number(1234.5);
```

## ベストプラクティス

### メッセージ構造

- 機能別・コンポーネント別にネストする
- 共通文言は `Common` / `Crud` などにまとめる

```json
{
  "Common": {
    "buttons": { "save": "保存", "cancel": "キャンセル" },
    "messages": { "loading": "読み込み中...", "error": "エラーが発生しました" }
  }
}
```

### バリデーションとの組み合わせ

```typescript
export function useFormSchema() {
  const t = useTranslations("Validation");
  return z.object({
    email: z.string().min(1, t("required")).email(t("invalidEmail")),
    password: z.string().min(8, t("passwordMinLength")),
  });
}
```

## 注意事項

AI がやりがちな間違い：

- JSON に重複キーがある
- JSON フォーマットが正しくない

JSON の正しいフォーマットを厳守すること。

## トラブルシューティング

- **`useTranslations can only be used in Client Components`** → Server では `getTranslations`、Client では `useTranslations`
- **翻訳が表示されない** → JSON フォーマット、キー階層、import パスを確認
- **型エラー** → `@types/node` と tsconfig のモジュール解決を確認
