# DataTable をどこでもクリック可能に

/admin/users などのAdminのCRUD のリスト画面で使われているDataTableだが、詳細ページ、編集ページへのアクセスが、一番右のセルにあるアイコンから行われるようになっている

これを以下のように変更する

- 行全体、どこをクリックしても詳細ページへリンクするようにする
- 一番右の列は廃止（削除）する
- 編集機能は詳細ページからのみアクセスできるように

## 追記: 実装計画（2026-03-21）

### 背景
- 現在の `DataTable` は [`src/components/organisms/DataTable/index.tsx`](../../src/components/organisms/DataTable/index.tsx) で共通実装されており、各行の右端に詳細 (`Eye`) / 編集 (`Pencil`) アイコンを持つアクション列を常に描画している。
- 実際の利用箇所は現時点では [`src/app/(site)/(authorized)/admin/users/page.tsx`](../../src/app/(site)/(authorized)/admin/users/page.tsx) のみだが、コンポーネント自体は管理画面の CRUD 一覧で再利用される前提の設計になっている。
- 詳細ページである [`src/app/(site)/(authorized)/admin/users/[id]/page.tsx`](../../src/app/(site)/(authorized)/admin/users/[id]/page.tsx) にはすでに編集導線が存在するため、一覧画面に編集アイコンを残す理由は弱く、同じ導線が二重化している。
- 一方で、単純に `DataTable` 全体を client component 化して `tr` クリックを実装すると、サーバー側で描画している一覧構造に不要なクライアント責務が増え、`Pagination` や既存テストへの影響範囲も広がる。
- そのため、要件である「行全体で詳細遷移」「右端アクション列の撤去」「編集導線を詳細ページに集約」を満たしつつ、共通コンポーネントとしての再利用性と保守性を落とさない実装方針を先に固定してから着手する必要がある。

### 方針とその理由
- **変更の主軸は `DataTable` 側に置き、一覧ページ個別の暫定対応はしない。**
  理由: 対象画面は現状 `/admin/users` のみでも、要件は「Admin の CRUD 一覧全体」に対する UX ルール変更であり、ページ側で個別にリンクを足すと将来の管理画面追加時に同じ修正が再発するため。
- **`DataTable` 自体は server component のまま維持し、行クリック責務だけを小さな client component に切り出す。**
  理由: 行全体クリックはクライアント側のイベント処理が必要だが、そのためだけにテーブル全体を client component 化するのは影響範囲が広い。行ラッパーだけを client 化すれば、要件を満たしつつ責務を局所化できる。
- **右端のアクション列は共通実装から削除し、一覧の主要操作は「行クリックで詳細へ」に一本化する。**
  理由: 一覧では詳細確認が第一操作であり、右端の小さなアイコンは発見性も押しやすさも低い。行全体を遷移ターゲットにすれば、情報密度と操作性の両方を改善できる。
- **編集導線は詳細ページにのみ残し、一覧では表示しない。**
  理由: 現在の詳細ページには編集ボタンがあり、一覧から直接編集に入る導線は必須ではない。まずは導線を詳細ページへ集約した方が UI の理解コストが低く、誤操作も減る。
- **将来 `type: "link"` などセル固有のインタラクションを持つ列が追加された場合は、そのセル内操作を優先し、行クリックは邪魔しない。**
  理由: 「行全体クリック」と「セル固有リンク」は衝突しやすい。今は `/admin/users` に該当列がないが、共通コンポーネントとしては将来の拡張を見越し、ネストしたリンクやボタンを壊さない前提で設計すべきである。
- **Storybook とテストを更新し、見た目だけでなく導線変更を回帰テストで固定する。**
  理由: 今回の変更は UI 上の導線変更であり、目視だけだと「右端列が復活する」「行クリックが効かなくなる」「詳細導線が消える」といった退行を見逃しやすいため。

#### 2. 実装方針の具体化
- [x] `DataTable` 本体は server component のまま維持し、行クリックだけを担当する client component を `src/components/organisms/DataTable/` 配下に追加する方針で構成を決める。
- [x] 新しい行ラッパー component の責務を定義する。少なくとも `href`、クリック時の遷移、キーボード操作、ネストされたインタラクティブ要素の除外判定、ホバー/フォーカス時の見た目制御をこの component に閉じ込める。
- [x] ネストされたインタラクティブ要素の除外対象を具体化する。少なくとも `a` / `button` / `input` / `select` / `textarea` / `summary` / `role="button"` / `data-row-click-ignore="true"` のような要素を候補として整理する。
- [x] 行クリック実装でアクセシビリティを損なわないよう、`tabIndex`、`role`、フォーカス可視化、スクリーンリーダー向けラベル方針を決める。
- [x] クリック可能行にした後も、既存のソートリンク、検索フォーム、ページネーション、sticky 先頭列、モバイル横スクロールが壊れないことを前提要件として明記する。
