# 管理画面の一覧のモバイル対応

## 追記: 実装計画（2026-03-12）

### 背景
- 現在の管理画面一覧は `src/components/organisms/DataTable/index.tsx` を使って描画しており、`/admin/users` ではページ全体の余白コンテナ内に `DataTable` が配置されている。
- モバイル幅では一覧自体をカード化したりレイアウトを切り替えたりするのではなく、`table` を維持したまま横スクロールで閲覧できることが要件である。
- 一方で現状は、モバイル時のテーブル表示領域が画面横幅いっぱいまで広がらず、左右に余白が残るため、一覧の情報量に対して表示効率が悪い。
- 今回の対象は共通 `DataTable` と、その利用中である `/admin/users` のみとし、検索・ソート・ページネーション・詳細/編集アクションはすべてモバイルでも維持する。

### 方針とその理由
- **`table` 構造は維持し、モバイル時だけスクロール領域を viewport 基準で広げる。**
  理由: 要件は「横スクロールをやめること」ではなく「横スクロール前提のまま、表示領域を画面幅まで使うこと」だから。カード化や key-value 表示への変更は要件外であり、既存のソート・列構造・アクセシビリティも崩しやすい。
- **対応の主軸は `DataTable` 側に置き、`/admin/users` 側の変更は最小限にとどめる。**
  理由: 今後ほかの管理画面でも同じ一覧コンポーネントを使う可能性が高く、モバイル時の横幅の扱いはコンポーネント側で統一した方が再利用性と保守性が高い。ただし、ページ側の余白構造が `DataTable` の全幅化を阻害する場合は `/admin/users` 側にも局所的な調整を入れる。
- **モバイルでは “フルブリードのスクロールコンテナ”、タブレット以上では従来のコンテナ幅を維持する。**
  理由: 問題はモバイル幅でのみ顕在化しており、`sm` 以上の既存レイアウトを崩す理由がない。base breakpoint でのみ viewport 幅に合わせ、`sm` 以上では従来の `max-w-*` や余白設計を維持するのが影響範囲を最小化できる。
- **検索フォーム・ページネーション・行アクションも、モバイル時の幅と操作性を合わせて確認する。**
  理由: テーブルだけを広げても、検索フォームが固定幅のまま不自然に余ったり、ページネーションや行末アクションがスクロールや折返しで壊れたりすると、実運用上は不完全だから。
- **見た目の確認だけで終わらせず、レイアウト用クラスの回帰をテストで固定する。**
  理由: 今回の変更は機能追加というよりレスポンシブ調整であり、後から崩れやすい。構造と class の意図をテストに残しておくと、再発防止に効く。
